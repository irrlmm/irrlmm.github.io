## The Legacy Experience

Before rebuilding the product, we mapped the existing mastering flow.

<Image
  client:load
  srcs={[]}
  alt="Legacy flow diagram showing sequential mastering workflow"
/>

```
Upload track
→ Choose parameters
→ Wait for preview (~5 min)
→ Listen
→ Restart if unsatisfied
→ Pay
```

This revealed several structural issues.

---

## Structural Problems

| Problem                      | Evidence                                                        |
| ---------------------------- | --------------------------------------------------------------- |
| Slow feedback loop           | Each preview generation took ~5 minutes                         |
| One-shot mental model        | Any change restarted the flow                                   |
| Unclear value before payment | Users could not compare results easily                          |
| High cognitive load          | Many controls with no immediate feedback                        |
| Low confidence threshold     | If the first result didn't feel right, users rarely tried again |

One thing _did_ work well:

✔ Audio quality was strong

The engine itself was not the problem.  
The interaction model was.

---

## Hypothesis

If users can generate and compare multiple mastering variants instantly in one session, confidence will increase and completion will increase.

---

## Exploration

We started with a radical prototype: chat-based mastering.

<Image
  client:load
  srcs={[]}
  alt="Prototype screens of chat-style mastering flow"
/>

The system suggested steps conversationally, and users selected responses through guided prompts.

However, testing revealed problems:

- High time to understand the interface
- Long scrolling sessions
- Hard to track state changes

The idea helped break assumptions, but it was not the right model.

---

## Breakthrough

We shifted to a comparison-first interaction model.

Instead of tweaking parameters sequentially, users could generate multiple mastering variants instantly and compare them.

<Image
  client:load
  srcs={[]}
  alt="Wireframe showing variant pills and comparison interface"
/>

Each pill represented a mastering result.

Users could switch between variants instantly without restarting the flow.

This removed iteration friction and externalized the decision.

---

## New Workflow

Before:

```
Upload → tweak parameters → wait → listen → repeat → pay
```

After:

```
Upload → generate variants → compare instantly → select → pay
```

Confidence moved before payment instead of after.

---

## Execution Constraints

The rebuild had to ship within two months.

Key decisions:

- Do not redesign the mediation layer between mastering and distribution
- Focus only on the core mastering experience
- Design features modularly so they could be enabled when backend work was ready

<Image
  client:load
  srcs={[]}
  alt="Scope definition artifact showing included vs excluded work"
/>

Execution followed a Lean UX model integrated with Scrum.

Design ran roughly half a sprint ahead of engineering.

## Results

The structural change delivered measurable improvements.

- +20% session completion
- 4.5 / 5 CSAT
- 25% adoption of reference mastering
- +21% increase in track output per user

Confidence increased because users could hear and compare results before paying.

The project did more than replace a vendor dependency.

It enabled:

- Vertical integration of mastering technology
- Reduced leakage to external AI tools
- A comparison-first interaction model aligned with market expectations
